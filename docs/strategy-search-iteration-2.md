## 发现的关键问题

### 1. DB 组件代码过期
源码更新后，MySQL 中的 MFile.data 没有同步。`ginkgo component edit --file` 调用 `file_service.update()` 实际未写入 data 字段。需要直接用 `file_crud().modify()` 更新。

### 2. FixedSizer 资金利用率低
- `volume=100` 在低价股（如紫金 11元）仅投入 1,100 元（1.1% 资金）
- `volume=10000` 在高价股（如赛力斯 89元）超出资金，订单被拒绝
- 需要根据股价合理设置 volume，或实现 RatioSizer

### 3. `file_service.update()` 不更新 data 字段
CLI 命令 `ginkgo component edit --file` 报告成功但实际未更新源码。必须用 `file_crud().modify(updates={'data': new_data})` 直接更新。

## 稳定性计算方法

稳定性 = 净值高于初始资金的交易日占比
```python
above_water = sum(1 for v in values if v >= initial)
stability = above_water / len(values) * 100
```

## 关键发现

1. **买入持有优于主动策略**: 在强趋势股票上，简单的买入持有大幅优于所有测试的主动策略
2. **资金利用率是关键**: volume 参数决定实际投入资金，直接影响年化收益
3. **DB 组件同步是痛点**: 源码更新后需手动 CRUD 更新 DB 中的组件代码
4. **RatioSizer 缺失**: 目前只有 FixedSizer，无法按资金比例自动计算仓位
