# CLI 策略构建探索报告

**日期**: 2026-05-07
**目标**: 通过 CLI 构建期望为正的稳定盈利策略（年化 15%+，稳定性 80%+）
**结论**: **无法通过当前 CLI 完成完整链路构建**

## 探索记录

### 尝试 1: MA5/20 交叉 + 000001.SZ（平安银行）
- 区间: 2024-01-01 ~ 2025-05-07
- 结果: **-159 元**（-0.16%），Sharpe -16.84，胜率 41.7%
- 信号 28 次，订单 25 次
- **分析**: 信号过于频繁，平安银行在此区间震荡（10~13），均线策略反复止损

### 尝试 2: MA20/60 交叉 + 000001.SZ
- 结果: **-329 元**（-0.33%），Sharpe -17.57
- 信号 10 次，订单 9 次
- **分析**: 减少信号频率反而更差，大周期滞后更严重

### 尝试 3: MA10/30 交叉 + 600519.SH（茅台）
- 结果: **-2446 元**（-2.4%），Sharpe -2.12，胜率 20%
- **分析**: 茅台在 2024-2025 区间下跌，均线死叉后持续亏损

### 尝试 4: MA5/20 + 300750.SZ（宁德时代）+ PositionRatioRisk
- 结果: **0 笔成交**，22 信号全部被风控拦截
- **根因**: `PositionRatioRisk` 无法在回测中获取当前价格 → `execution_price=0` → 全部 ORDERBLOCKED
- 这是一个严重的 bug

### 尝试 5: MA5/20 + 300750.SZ（宁德时代）+ NoRisk
- 结果: **+581 元**（+0.58%），Sharpe -1.93，胜率 33%
- **分析**: 虽然微盈利，但 Sharpe 极差，远不达标

### 尝试 6: 买入持有 + 300750.SZ（宁德时代）
- 策略: RandomSignalStrategy(buy=1.0, sell=0.0, max_signals=5, volume=30)
- 区间: 2024-01-01 ~ 2025-05-07
- 结果: **+10,410 元**（+10.4%），年化 5.2%，Sharpe 0.28，最大回撤 8.9%
- **分析**: 仓位太小（仅 4.25% 资金利用），资金效率低

### 尝试 7: 重仓买入持有 + 601899.SH（紫金矿业）
- 策略: RandomSignalStrategy(buy=1.0, sell=0.0, max_signals=10, volume=500)
- 区间: 2024-01-01 ~ 2025-05-07
- 结果: **+25,865 元**（+25.9%），年化 12.5%，Sharpe 0.60，最大回撤 18.1%
- **分析**: 最接近目标的结果，但仍低于 15% 年化要求
- **本质问题**: 这不是"策略"，只是选了一只涨的股票持有

### 尝试 8: 紫金矿业更长周期（2023-01 ~ 2025-05）
- 结果: **+30,740 元**（+30.7%），年化 8.2%，Sharpe 0.40，最大回撤 17.5%
- **分析**: 周期越长年化越低，说明早期买入价格不够低

### 尝试 9: 紫金矿业波段（2024-06 ~ 2025-05）
- 结果: **+3,770 元**（+3.8%），年化 2.8%，Sharpe 0.09，最大回撤 20.9%
- **分析**: 错过了最好的建仓时机

### 尝试 10: 自定义突破策略 + 紫金矿业（第一版）
- 策略: 自写 N日突破策略（通过 Python 直接更新数据库组件源码）
- 结果: **0 笔成交**，8 信号但 0 订单
- **根因**: `create_signal()` 缺少 `business_timestamp` 参数，导致 T+1 检查失败
- **发现**: 可以通过 `FileService.update()` / CRUD.modify() 更新组件源码

### 尝试 11: 自定义突破策略（修复 business_timestamp）+ 紫金矿业
- 结果: **+741 元**（+0.74%），年化 0.38%，Sharpe -1.50，胜率 33%
- 信号 8 次，订单 7 次
- **分析**: 突破策略在 A 股假突破频繁的环境下表现不佳

### 尝试 12: 买入持有 + 601127.SH（赛力斯）
- 策略: RandomSignalStrategy(buy=1.0, sell=0.0, max_signals=1, volume=2000)
- 区间: 2024-01-01 ~ 2025-05-07
- 结果: **+75,372 元**（+75.4%），年化 **33.4%**，Sharpe 0.82，最大回撤 30.5%
- **年化超额达标**，但最大回撤 30.5% → 稳定性仅 69.5%（需 80%）

### 尝试 13: 自定义趋势跟踪+8%回撤止损 + 赛力斯
- 策略: 自写 MA20 金叉买入 + 8% 峰值回撤止损
- 结果: **+7,430 元**（+7.4%），年化 3.75%，Sharpe 0.14，最大回撤 26.7%，胜率 25%
- **分析**: 止损过紧，反复被震荡洗出，错失主升浪

### 尝试 14: 自定义趋势跟踪+15%回撤止损 + 赛力斯
- 结果: **-3,039 元**（-3%），年化 -1.57%，Sharpe -0.10，最大回撤 33.3%，胜率 33%
- **分析**: 宽止损更差，持有期间承受更大回撤后止损卖出

## 发现的系统性问题

### 1. 策略组件质量极低

| 策略 | 状态 | 问题 |
|------|------|------|
| moving_average_crossover | **可用但有 bug** | `_ma_states` 按覆盖而非按 code 存储，多股场景崩溃 |
| mean_reversion | **不可用** | RSI 计算代码有语法错误（`closes - closes` 而非 `closes[i] - closes[i-1]`），gains/losses 空列表 |
| trend_follow | **不可用** | `symbols=,` 语法错误，`portfolio_info["positions"]` 类型假设错误 |
| dual_thrust | **不可用** | 同样 `symbols=,` 语法错误 |
| momentum | **不可用** | 需要多只股票做截面排名，且依赖 data_feeder 接口 |
| scalping | **空壳** | `cal()` 直接返回 `[]` |
| price_action | **空壳** | 只有注释，无实现 |
| volume_activate | **未验证** | |
| ratio_sizer | **空壳** | 只有头注释，无实现 |
| atr_sizer | **未验证** | |

### 6. 参考的已有 Portfolio 组装模式

从 `focus_v7`、`winner_v1` 等历史 portfolio 发现：
- 全部使用 `random_signal_strategy`（buy=0.99, sell=0.0），即"只买不卖"
- 全部使用 `no_risk`，因为 `PositionRatioRisk` 在回测中失效
- 本质上是在"赌"某只股票会涨，不是真正的量化策略
- 这说明**项目自身也未能通过 CLI 构建出有效的量化策略**

### 2. PositionRatioRisk 风控在回测中完全失效

**现象**: 所有订单被 `ORDERBLOCKED`，`execution_price=0`

**根因**: PositionRatioRisk.cal() 需要获取当前价格来计算持仓市值。在回测引擎中，Market Order 的 `limit_price=0`，且 PositionRatioRisk 无法从 event 中获取价格，导致判断价格无效 → 阻断所有订单。

**影响**: 无法使用任何风控组件进行回测。只能用 `NoRisk`。

### 3. CLI 缺少组件编辑功能

- `ginkgo component create` 只生成基础 MA 交叉模板
- **没有 `ginkgo edit` 命令**，无法编辑已创建组件的源码
- 模板本身也有 bug（`_closes = deque()` 而非 `_closes[code] = deque()`）
- 无法通过 CLI 创建自定义策略逻辑

### 4. 数据探索工具不完善

- `ginkgo data get day` → "Bar data get not yet implemented"
- `ginkgo data get sources` → "Data sources functionality not yet implemented"
- 无法查看哪些股票有 K 线数据
- 无法通过 CLI 判断数据质量

### 5. Portfolio/Backtest 操作不便

- `portfolio list` UUID 被截断，创建后需手动记录完整 ID
- `portfolio get <truncated_uuid>` 无法匹配（需完整 UUID 或 name）
- `backtest cat` 缺少更多分析维度（如月度收益、最大连续亏损金额）
- 无法对比多次回测结果

## 最终结论

经过 14 次系统性回测（含 3 个自定义策略），**确认无法同时达到年化 15%+ 和稳定性 80%+ 的目标**：

| 策略 | 股票 | 年化 | 最大回撤 | 稳定性 | 达标 |
|------|------|------|---------|--------|------|
| MA5/20 交叉 | 000001.SZ | -0.16% | 0.31% | - | ✗ |
| MA20/60 交叉 | 000001.SZ | -0.33% | 0.40% | - | ✗ |
| MA10/30 交叉 | 600519.SH | -2.4% | - | - | ✗ |
| 买入持有 | 300750.SZ | 5.2% | 8.9% | 91% | ✗ (收益不足) |
| 重仓买入持有 | 601899.SH | **12.5%** | 18.1% | 82% | ✗ (收益不足) |
| **买入持有** | **601127.SH** | **33.4%** | **30.5%** | **69.5%** | **✗ (稳定性不足)** |
| 突破策略(自定义) | 601899.SH | 0.38% | 2.7% | - | ✗ |
| 趋势+8%止损(自定义) | 601127.SH | 3.75% | 26.7% | 73% | ✗ |
| 趋势+15%止损(自定义) | 601127.SH | -1.57% | 33.3% | 67% | ✗ |

**核心矛盾**: 高收益需要持仓不动（买入持有 33.4%），但持仓不动必然承受大回撤（30.5%）；加了止损控制回撤，但止损会洗出仓位导致收益大幅下降。两者在单股+日线+无对冲的约束下不可兼得。

### 关键发现：组件源码更新方式

`ginkgo edit` 命令不存在，但可以通过 Python 直接更新数据库中的组件源码：
```python
from ginkgo.data.containers import container
crud = container.file_crud()
crud.modify(filters={'uuid': '<component_uuid>'}, updates={'data': source_bytes})
```
**建议将此功能包装为 `ginkgo component edit` CLI 命令。**

## 无法达标的原因总结

1. **只有一种可用策略**（MA 交叉），且该策略在 A 股近两年震荡市中天然不具备 alpha
2. **风控系统在回测中完全失效**，无法控制回撤
3. **无法创建自定义策略**，无法实现任何有 alpha 的逻辑
4. **数据工具缺失**，无法选股、无法评估数据质量

## 建议改进方向

1. **修复 PositionRatioRisk**：在回测模式下从 order 或 event 获取价格，而非依赖独立的 price query
2. **实现 `ginkgo component edit`**：允许通过 CLI 编辑组件源码
3. **修复已有策略**：至少修复 mean_reversion 的 RSI 计算和 trend_follow 的语法错误
4. **实现 `ginkgo data get day`**：允许查看可用 K 线数据
5. **改进 backtest 结果展示**：增加月度收益表、交易明细、持仓历史
